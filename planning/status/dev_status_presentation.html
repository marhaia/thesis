Kontext

Forschungsfrage & Ansatz

"Does the integration of cognitive predictive metrics into the AIM platform improve the computational evaluation of GUIs?"

Problem

AIM analysiert visuelle Eigenschaften, nicht Nutzerverhalten. Eine GUI kann visuell ordentlich sein und trotzdem hohe kognitive Belastung erzeugen.

Lösung: Zweistufige Pipeline

Stage 1 → Feature-Extraktion aus Screenshot
(visuelle Komplexität + Saliency + Kognition)

Stage 2 → Cognitive Load Index (0–100)
(HCEye-Koeffizienten + Task-Kontext)

Stage 1c

Jokinen 2020 Kognitives Suchmodell

Modell

Adaptive Feature Guidance (AFG)
Jokinen et al. (2020). IJHCS, 136, 102376

Input: UI-Elemente (aus element_detector.py) + UMSI++ Saliency Map
Prozess: Monte-Carlo-Simulation (50–500 Trials/Element)
Cognitive Modell: EMMA + VSTM (Visual Short-Term Memory)

Output: Predicted Visual Search Time pro Element

Endpoint: POST /api/search-time

Status

✅ Vollständig implementiert
✅ UMSI++ als Bottom-Up-Signal integriert
✅ Element-Detektor für UI-Typen vorhanden

Limitation

⚠️ Parameter sind Literaturwerte (EMMA, VSTM), nicht empirisch kalibriert

⚠️ Element-Detektor ist regelbasiert — ML-basierter Detektor würde Genauigkeit signifikant verbessern

⚠️ Keine Sensitivitätsanalyse der Monte-Carlo-Parameter

Kodiert Aufgaben-Kontext für den Cognitive-Load-Adjustment:

• Aufgabentyp: navigation, search, data_entry, decision, monitoring
• Spezifität: High Specificity → niedrigere Last

Honest Assessment

Aktuelle Limitationen

1. Kein empirisches Training

Stage 2 läuft im rule-based Modus. Die Regression ist ein Gerüst ohne trainierte Parameter. Default-Path ist rule-based, nicht datengetrieben.

2. Limitierter Element-Detektor

Jokinen-Modell-Genauigkeit hängt von UI-Element-Erkennung ab. Aktuell regelbasiert → systematische Fehler bei komplexen Layouts.

3. Task-Gewichte ordinal

task_descriptor.py verwendet Literatur-basierte Ordinalwerte (Wickens, Paas). Müssten durch Nutzerstudie kalibriert werden.

4. UMSI++ auf CPU

~5–15 Sek/Bild auf M4 Mac. GPU-Support möglich aber nicht eingerichtet. Bei Nutzerstudie (N≈30–35) kein Problem, bei größerem Datensatz bottleneck.

5. Fehlende stat. Validierung

Kein ICC (Intra-Class Correlation), kein Cronbach's Alpha für Feature-Stabilität. Wichtig für Kapitel "Construct Validity".

Was gut ist

✅ Alle theoretischen Bausteine sind implementiert
✅ Default-Pfad gibt plausible Scores
✅ System läuft end-to-end
✅ API ist vollständig verwendbar
